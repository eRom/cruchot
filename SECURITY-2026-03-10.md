## Vuln 1 — CRITICAL: Lecture arbitraire de fichiers via `local-image://` — `src/main/index.ts:14-23`

* **Sévérité:** Critical
* **Catégorie:** `path_traversal` / `arbitrary_file_read`
* **Confiance:** 9/10
* **Statut:** ✅ Corrigé (2026-03-10)
* **Description:** Le handler du protocole custom `local-image://` sert n'importe quel fichier du système sans aucune validation de chemin. Le scheme est enregistré avec `bypassCSP: true`, ce qui neutralise toute protection CSP. Un renderer compromis (XSS via contenu LLM) peut lire `/etc/passwd`, `~/.ssh/id_rsa`, ou tout fichier accessible au processus Electron.
* **Code vulnérable:**
  ```typescript
  protocol.handle('local-image', (request) => {
    const filePath = decodeURIComponent(request.url.replace('local-image://', ''))
    return net.fetch(pathToFileURL(filePath).href)  // Aucune validation !
  })
  ```
* **Scénario d'exploitation:** Un LLM renvoie du contenu markdown crafté qui déclenche du XSS (voir Vuln 4). Le script injecté appelle `fetch('local-image:///Users/recarnot/.ssh/id_rsa')` et exfiltre le contenu via `window.api` ou un canal réseau.
* **Recommandation:** Valider que le chemin résolu commence par un répertoire autorisé (`userData/images`, `userData/attachments`, workspace root). Retirer `bypassCSP: true` et ajouter `local-image:` à `img-src` dans la CSP.
  ```typescript
  protocol.handle('local-image', (request) => {
    const filePath = decodeURIComponent(request.url.replace('local-image://', ''))
    const resolved = path.resolve(filePath)
    const allowedDirs = [
      path.join(app.getPath('userData'), 'images') + path.sep,
      path.join(app.getPath('userData'), 'attachments') + path.sep,
    ]
    if (!allowedDirs.some(dir => resolved.startsWith(dir))) {
      return new Response('Forbidden', { status: 403 })
    }
    return net.fetch(pathToFileURL(resolved).href)
  })
  ```

## Vuln 2 — CRITICAL: Exécution arbitraire via `files:openInOS` — `src/main/ipc/files.ipc.ts:124-129`

* **Sévérité:** Critical
* **Catégorie:** `command_execution`
* **Confiance:** 9/10
* **Statut:** ✅ Corrigé (2026-03-10)
* **Description:** Le handler IPC `files:openInOS` appelle `shell.openPath(filePath)` avec pour seule validation un `typeof === 'string'`. Aucune restriction de répertoire ni d'extension. Un renderer compromis peut ouvrir n'importe quel fichier ou application (`.app`, `.command`, `.sh`), menant à une exécution de code arbitraire.
* **Scénario d'exploitation:** Via XSS dans le renderer, l'attaquant appelle `window.api.filesOpenInOS('/Applications/Terminal.app')` ou pointe vers un script malveillant déposé sur le filesystem.
* **Recommandation:** Restreindre aux répertoires légitimes (`userData/images`, `userData/attachments`, workspace root) et valider l'extension contre une allowlist.
  ```typescript
  const SAFE_DIRS = [imagesDir, attachmentsDir]
  const resolved = path.resolve(filePath)
  if (!SAFE_DIRS.some(dir => resolved.startsWith(dir + path.sep))) {
    throw new Error('Path not allowed')
  }
  ```

## Vuln 3 — CRITICAL: Path Traversal sur `backup:restore` et `backup:delete` — `src/main/services/backup.service.ts:58-72,99-102`

* **Sévérité:** Critical
* **Catégorie:** `path_traversal`
* **Confiance:** 9/10
* **Statut:** ✅ Corrigé (2026-03-10)
* **Description:** `restoreBackup(backupPath)` copie n'importe quel fichier fourni par le renderer pour écraser la base SQLite de l'app (`copyFileSync(backupPath, dbPath)`). `deleteBackup(backupPath)` appelle `unlinkSync` sur n'importe quel chemin. La seule validation est `z.string().min(1)` — aucune vérification que le chemin est dans le répertoire backups.
* **Scénario d'exploitation:**
  - **Restore:** Un renderer compromis appelle `backup:restore` avec `/tmp/malicious.db` → la base de l'app est remplacée par une base SQLite piégée (injection de données, XSS stored via messages).
  - **Delete:** Appel avec `/Users/recarnot/.zshrc` → fichier supprimé irréversiblement (`unlinkSync`, pas `trash`).
* **Recommandation:** Valider que le chemin résolu est dans le répertoire backups.
  ```typescript
  const resolved = path.resolve(backupPath)
  const backupsDir = getBackupsDir()
  if (!resolved.startsWith(backupsDir + path.sep)) {
    throw new Error('Invalid backup path')
  }
  ```

## Vuln 4 — HIGH: XSS via Mermaid `securityLevel: 'loose'` + `dangerouslySetInnerHTML` — `src/renderer/src/components/chat/MermaidBlock.tsx:15,82`

* **Sévérité:** High
* **Catégorie:** `xss`
* **Confiance:** 8/10
* **Statut:** ✅ Corrigé (2026-03-10)
* **Description:** Mermaid est initialisé avec `securityLevel: 'loose'` (désactive les protections XSS built-in) et le SVG rendu est injecté via `dangerouslySetInnerHTML` sans sanitization. Le contenu Mermaid provient des réponses LLM. Un diagramme Mermaid crafté peut contenir des handlers d'événements SVG (`onclick`, `onerror`) ou des éléments `<foreignObject>` avec du JavaScript.
* **Scénario d'exploitation:** Via prompt injection, le LLM génère un bloc mermaid contenant `click nodeA callback "<img src=x onerror=fetch('local-image:///etc/passwd').then(...)>"`. Avec `securityLevel: 'loose'`, Mermaid ne sanitize pas ce contenu, et `dangerouslySetInnerHTML` l'injecte dans le DOM. Ceci active la chaîne d'exploitation de la Vuln 1.
* **Recommandation:** Changer `securityLevel` à `'strict'` et sanitizer le SVG avec DOMPurify avant injection.
  ```typescript
  import DOMPurify from 'dompurify'

  mermaid.initialize({
    securityLevel: 'strict',  // était 'loose'
    // ...
  })

  // Avant injection :
  dangerouslySetInnerHTML={{ __html: DOMPurify.sanitize(svgOutput) }}
  ```

## Actions Immédiates

1. **`local-image://`** : Ajouter un allowlist de répertoires (`userData/images`, `userData/attachments`, workspace root). Retirer `bypassCSP: true`.
2. **`files:openInOS`** : Restreindre `shell.openPath` aux répertoires connus + allowlist d'extensions.
3. **`backup:restore/delete`** : Valider `path.resolve(backupPath).startsWith(getBackupsDir() + path.sep)`. Remplacer `unlinkSync` par `trash`.
4. **Mermaid** : Passer à `securityLevel: 'strict'`. Ajouter DOMPurify avant `dangerouslySetInnerHTML`.
